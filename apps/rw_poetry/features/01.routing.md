
# Feature: Static-Server-Compatible Routing

## Status
Resolved — Option A (hash routing) implemented. See `src/routing.rs` and spec section 4.3.

## Problem Statement

The app currently uses path-based (History API) client-side routing via `leptos_router`. Under this model, navigating to `/readings` or `/readings/:recording_id` causes the browser to request those paths from the server. The server must be configured to return `index.html` for all such requests (a "catch-all" or "SPA fallback" rule) so the WASM app can boot and handle routing itself.

This is standard practice on full-featured hosting platforms (Netlify, Vercel, Cloudflare Pages), but it is a non-trivial configuration requirement on minimal or shared static hosting environments — cheap Apache/nginx shared hosts, simple Python `http.server` instances, GitHub Pages in subdirectory mode, or any host that only serves files exactly as they exist on disk.

The goal is to enable the app to be dropped into any static file server and work without any server-side routing configuration.

---

## Current State

- **Router type:** `leptos_router::Router` (History API / path-based)
- **Routes defined in:** `src/app.rs`
- **Affected routes:**
  - `/` — Reader view
  - `/readings` — Recordings list
  - `/readings/:recording_id` — Recording detail

A direct browser load of `/readings` on a plain static server returns a 404 because no such file exists on disk.

---

## Options

### Option A — Hash-Based Routing (Recommended)

Replace the `Router` component with `HashRouter` from `leptos_router`. All route paths are prefixed with `#`, so `/readings` becomes `/#/readings` and `/readings/:id` becomes `/#/readings/:id`.

The fragment portion of a URL (`#...`) is never sent to the server — it is handled entirely by the browser. A static server receives only a request for `/index.html` (or `/`) on every navigation, which it can always fulfill.

**How it works in Leptos 0.8:**

```rust
use leptos_router::components::{HashRouter, Route, Routes};
use leptos_router::path;

view! {
    <HashRouter>
        <TopBar />
        <Routes fallback=|| view! { <p>"Page not found."</p> }>
            <Route path=path!("/") view=ReaderView />
            <Route path=path!("/readings") view=RecordingsListView />
            <Route path=path!("/readings/:recording_id") view=RecordingDetailView />
        </Routes>
    </HashRouter>
}
```

The route definitions themselves do not change. Only the outer `Router` component is swapped.

**Impact on existing code:**
- `use_params_map()`, `use_query_map()`, `<a href="...">` links, and `leptos_router::hooks::navigate()` all continue to work unchanged under `HashRouter`.
- Hardcoded `href` strings in the UI (e.g. `href="/readings"`, `href=format!("/readings/{}", id)`) must be updated to use the `#`-prefixed form: `href="#/readings"`, `href=format!("#/readings/{}", id)`. Alternatively, use `leptos_router`'s `<A>` component which handles this automatically.

**Pros:**
- Works on any static server with zero configuration.
- Minimal code change — one component swap plus link updates.
- Full browser history (back/forward) still works.
- `use_params_map()` and `use_query_map()` still work correctly.

**Cons:**
- URLs are less clean (`/#/readings` vs `/readings`).
- Fragment routing is considered unfashionable; some link-sharing contexts may strip or mishandle the `#` portion.
- Server-side rendering is not possible (not a concern for this CSR-only app).

---

### Option B — Static Server Fallback Configuration

Keep path-based routing and configure the server to serve `index.html` for all unmatched paths.

**Apache** (`.htaccess`):
```apache
<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteRule ^ index.html [L]
</IfModule>
```

**nginx:**
```nginx
location / {
  try_files $uri /index.html;
}
```

**Pros:**
- URLs stay clean.
- No code changes required.

**Cons:**
- Requires server configuration access — not available on all cheap static hosts.
- Does not work with a simple Python `http.server`, `npx serve` (without flags), or file:// protocol.
- Each new deployment target needs to be configured separately.
- Does not solve the problem for the stated constraint ("drop into any static server").

---

### Option C — 404.html Fallback Trick

Some static hosts (notably GitHub Pages) serve a file named `404.html` when a path is not found. Copying `index.html` to `404.html` causes the app to load even on unrecognised paths, after which the client-side router handles the route.

**Trunk configuration to copy at build time:**
```html
<!-- In index.html or via Trunk.toml post-build hook -->
```
Or a simple post-build script:
```bash
cp dist/index.html dist/404.html
```

**Pros:**
- No code changes.
- Works on GitHub Pages without any other config.

**Cons:**
- Host-specific — only works on platforms that implement the `404.html` convention.
- The browser briefly receives a 404 HTTP status before the app boots, which can interfere with some link unfurling / SEO tools (not a concern here, but worth noting).
- Not portable; does not solve the general "any static server" requirement.

---

### Option D — Signal-Based View Switching (No URL Routing)

Remove `leptos_router` entirely and replace the router with a top-level signal that tracks which view is active. Navigation is a signal write; the top-level component switches on it.

```rust
#[derive(Clone, PartialEq)]
enum View {
    Reader,
    RecordingsList,
    RecordingDetail(String), // recording_id
}

let current_view = RwSignal::new(View::Reader);
```

**Pros:**
- Zero server requirements.
- No URL-related code at all.

**Cons:**
- Browser back/forward button does not work.
- Deep-linking and bookmarking are impossible.
- Sharing a link to a specific recording detail is not possible.
- Significant refactor: all navigation, `use_params_map()`, and `use_query_map()` calls must be replaced.
- Eliminates the `← Read this poem` link behavior that passes `poem_id` via query param.

This option is not recommended except as a last resort.

---

## Recommendation

**Option A (HashRouter)** is the right choice for this app. The constraints are clear: zero server configuration, works everywhere, minimal disruption to existing code. The URL aesthetic trade-off (`/#/readings`) is minor for a personal local-first app that is not indexed by search engines and where links are not shared externally.

The implementation is a contained change in `src/app.rs` and a sweep of hardcoded `href` strings in the UI components.
