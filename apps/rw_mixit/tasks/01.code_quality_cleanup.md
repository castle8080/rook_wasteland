## Overview

## Issue 1 — Recoverable errors cause panics (HIGHEST PRIORITY)

### Problem

`src/audio/loader.rs` panics on failures that can legitimately occur at runtime:
a corrupted audio file, an unsupported format, or a `FileReader` that hasn't
fired yet. A panic in WASM terminates the entire module — the user's tab is dead
with no explanation.

Specific callsites:
- `FileReader::new().expect("FileReader::new")` — should be infallible, but
  message gives no reason.
- `reader.read_as_array_buffer(&file).expect(...)` — can fail if the File handle
  is stale (e.g. after a system sleep/wake on some browsers).
- `decode_audio_data(...)` returns a `Promise` that rejects on corrupt or
  unsupported audio. The rejection is currently unhandled — the promise is
  `.expect()`-ed at the call site and the rejection path is silently swallowed.
- `reader.result().expect("FileReader.result")` — called inside the `onload`
  callback, which is correct; message should explain the invariant.

### Fix

`load_audio_file` should propagate errors to the UI rather than panic.

1. Change `load_audio_file` to surface errors via a dedicated error signal on
   `DeckState` (e.g. `load_error: RwSignal<Option<String>>`), or accept an
   `on_error: impl Fn(String)` callback, so the deck component can display a
   user-visible message (e.g. "Could not decode file — unsupported format").
2. Wrap the `decodeAudioData` promise rejection explicitly. `wasm_bindgen_futures`
   `JsFuture` resolves to `Result<JsValue, JsValue>`; the `Err` branch must be
   matched and handled, not ignored.
3. Replace surviving `.expect()` calls in loader with messages that explain the
   invariant (see Issue 2 below for the pattern).
4. Log all handled errors to the browser console via `web_sys::console::error_1`
   as well as surfacing them in the UI.

## Issue 2 — `.expect()` messages describe what, not why

### Problem

97 `.expect()` calls exist. Most correctly use `.expect()` on genuinely
infallible operations (Web Audio node creation, canvas transforms). However the
messages uniformly describe *what* is called rather than *why it cannot fail*:

```rust
// Current — not useful when reading a panic in the console
ctx.create_gain().expect("create_gain pre_gain")
ctx.arc(cx, cy, r, 0.0, TAU).expect("platter: disc arc")
```

A good message explains the invariant so a future reader (or a bug report) knows
whether the panic means a browser bug, a code bug, or an environment problem.

### Fix

Audit all `.expect()` callsites and rewrite messages to state the invariant:

```rust
// Pattern for Web Audio node creation
ctx.create_gain()
    .expect("AudioContext.createGain() is infallible on a live context (Web Audio spec §1.6)")

// Pattern for canvas operations
ctx.arc(cx, cy, r, 0.0, TAU)
    .expect("arc() only fails if radius < 0; r is computed as canvas_width/2 which is always > 0")

// Pattern for DOM access
web_sys::window()
    .expect("window is always present in a browser WASM context")
```

Files to audit: `deck_audio.rs` (~45 calls), `loader.rs` (~6 calls),
`platter_draw.rs` (~9 calls), `raf_loop.rs` (~4 calls), `context.rs` (~2 calls).

## Issue 3 — `.unwrap()` used outside tests

### Problem

9 `.unwrap()` calls appear in non-test production code. Every one should be
either `.expect("reason")` (if truly infallible) or proper error handling (if
not).

| File | Approximate location | Call | Action |
|---|---|---|---|
| `raf_loop.rs` | RAF closure borrow | `f.borrow().as_ref().unwrap()` | `.expect("closure is initialised before rAF fires")` |
| `raf_loop.rs` | RAF closure borrow | `g.borrow().as_ref().unwrap()` | Same |
| `platter_draw.rs` | Offscreen canvas creation | `doc.create_element("canvas").unwrap()` | `.expect("createElement('canvas') is infallible in all supported browsers")` |
| `platter_draw.rs` | Attribute set | `.set_attribute("width", "240").unwrap()` | `.expect("set_attribute on a freshly created element cannot fail")` |
| `platter_draw.rs` | Attribute set | `.set_attribute("height", "240").unwrap()` | Same |
| `header.rs` | Window/document chain | `window().unwrap()` | `.expect("window always present in browser WASM")` |
| `header.rs` | Window/document chain | `.document().unwrap()` | `.expect("document always present when window exists")` |
| `context.rs` | Borrow on known-Some value | `borrow.as_ref().unwrap()` | Refactor to use the `if let` branch above, eliminating the unwrap |

### Fix

Replace each with `.expect("…")` using the message pattern from Issue 2.
The `context.rs` case should be restructured to eliminate the unwrap entirely —
the surrounding `if` already checks `is_some()` so the `.unwrap()` is
redundant; refactor to `if let Some(ctx) = borrow.as_ref()`.

## Issue 4 — Bare primitives; illegal states are representable

### Problem

All domain values are bare `f32` or `f64` with no type-level range enforcement:

- `loop_in: RwSignal<f64>` and `loop_out: RwSignal<f64>` can be set so that
  `loop_in >= loop_out`. This invalid state is only caught by a defensive check
  in `raf_loop.rs` at draw time — not at the point of assignment.
- `crossfader: RwSignal<f32>` has no enforcement of `[0.0, 1.0]`.
- `master_volume` / `volume` signals have no `[0.0, 1.0]` enforcement.
- BPM is validated by an `assert!` inside `bpm.rs` rather than by a type.
- `zoom_level: RwSignal<u8>` accepts any `u8`; valid values are `1, 2, 4, 8`.

### Fix

Introduce newtypes for the domain values that carry constraints:

```rust
// src/state/types.rs  (new file)

/// Playback position in seconds. Always >= 0.0.
#[derive(Clone, Copy, Debug, Default, PartialEq, PartialOrd)]
pub struct Seconds(f64);
impl Seconds {
    pub fn new(v: f64) -> Self {
        debug_assert!(v >= 0.0, "Seconds must be non-negative");
        Self(v.max(0.0))
    }
    pub fn get(self) -> f64 { self.0 }
}

/// Normalised gain/volume in [0.0, 1.0].
#[derive(Clone, Copy, Debug, PartialEq, PartialOrd)]
pub struct NormGain(f32);
impl NormGain {
    pub fn new(v: f32) -> Self { Self(v.clamp(0.0, 1.0)) }
    pub fn get(self) -> f32 { self.0 }
}

/// A valid loop region: guarantees out > in.
#[derive(Clone, Copy, Debug)]
pub struct LoopRegion {
    pub in_secs: Seconds,
    pub out_secs: Seconds,
}
impl LoopRegion {
    pub fn new(in_secs: Seconds, out_secs: Seconds) -> Option<Self> {
        if out_secs > in_secs { Some(Self { in_secs, out_secs }) } else { None }
    }
}

/// Zoom level — only powers of two in [1, 8] are valid.
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum ZoomLevel { X1 = 1, X2 = 2, X4 = 4, X8 = 8 }
impl ZoomLevel {
    pub fn zoom_in(self) -> Self { ... }
    pub fn zoom_out(self) -> Self { ... }
}
```

**Scope note:** This is the most invasive change — it touches signal declarations
in `state/deck.rs`, `state/mixer.rs`, and every site that reads/writes those
signals. Do it module by module rather than all at once. Start with
`LoopRegion` (highest risk of real bugs) and `ZoomLevel` (trivial enum, zero
risk), then `NormGain`, then `Seconds`.

## Issue 5 — No Clippy deny list

### Problem

`lib.rs` has no `#![warn(...)]` or `#![deny(...)]` attribute block. Without
this, new `.unwrap()` or unreviewed panics added in future milestones will
compile silently.

The three `#[allow(dead_code)]` attributes on state structs (`state/deck.rs`,
`state/mixer.rs`) are almost certainly Clippy false positives — the fields are
all accessed through Leptos signals which Clippy's dead-code pass doesn't trace.
They should be replaced with a targeted suppression that documents this.

### Fix

Add to the top of `src/lib.rs`:

```rust
#![warn(clippy::unwrap_used)]
#![warn(clippy::todo)]
// expect_used is intentionally left as warn-only (not deny) because .expect()
// with a good message is acceptable for infallible Web Audio / DOM operations.
// Reviewers should audit new expect() callsites manually.
```

Replace the blanket `#[allow(dead_code)]` on state structs with:

```rust
// Leptos RwSignal fields are read via signal getters in view! closures;
// rustc's dead-code pass does not trace through the reactive macro boundary.
#[allow(dead_code)]
```

(The comment is the important part — make the suppression self-explaining.)

## Work Order

Execute issues in this order to keep each commit reviewable and the build green:

1. **Issue 3** — Replace `.unwrap()` with `.expect()`. Pure find-and-replace;
   zero behaviour change; lowest risk. One commit per file.
2. **Issue 2** — Improve `.expect()` messages. Same risk profile. Work file by
   file: `deck_audio.rs`, `platter_draw.rs`, `loader.rs`, `raf_loop.rs`,
   `context.rs`.
3. **Issue 5** — Add Clippy deny list. Run `cargo clippy --target
   wasm32-unknown-unknown -- -D warnings` after and fix anything that surfaces.
4. **Issue 1** — Add error signal to `DeckState`, wire `load_audio_file` to
   return/surface errors, handle `decodeAudioData` rejection. This is the only
   change that touches component behaviour (the UI gains an error display).
5. **Issue 4** — Introduce newtypes. Do `ZoomLevel` first (enum, no signal
   changes), then `LoopRegion`, then `NormGain`, then `Seconds`. Each as its
   own commit.

## Implementation Notes

### Issue 1 — Loader error handling
- Added `load_error: RwSignal<Option<String>>` to `DeckState`.
- `read_file_as_array_buffer` now returns `Result<ArrayBuffer, String>`;
  an `onerror` closure on `FileReader` rejects the promise with a descriptive
  message instead of panicking.
- `load_audio_file` clears the error signal at start, then `match`es on both
  the FileReader result and the `JsFuture::from(decode_audio_data).await`
  result. Errors are written to `state.load_error` and logged via
  `web_sys::console::error_1`.
- The `Deck` component view now renders a `<div class="load-error">` whenever
  `state.load_error` is `Some`.

### Issue 2 — `.expect()` message quality
- Rewrote ~50 `.expect()` messages across `deck_audio.rs`, `mixer_audio.rs`,
  `waveform_draw.rs`, `platter_draw.rs`, `raf_loop.rs`, `context.rs`,
  `loader.rs`. All messages now state the invariant that makes the call
  infallible rather than simply naming the call.

### Issue 3 — `.unwrap()` outside tests
- All 9 `.unwrap()` calls in production code replaced with `.expect()`.

### Issue 4 — `ZoomLevel` enum
- Added `ZoomLevel { X1=1, X2=2, X4=4, X8=8 }` to `state/deck.rs` with
  `zoom_in()`, `zoom_out()`, `factor()`, and `Display` implementations.
- `DeckState.zoom_level` changed from `RwSignal<u8>` to `RwSignal<ZoomLevel>`.
- `WaveformCache.zoom` changed from `u8` (sentinel 0) to `Option<ZoomLevel>`
  (sentinel `None`). `draw_peaks_to_canvas` takes `ZoomLevel` and uses
  `zoom.factor()` in the visible-count calculation.
- `ZoomControls` component updated to use `zoom_in()` / `zoom_out()` methods,
  eliminating the manual `if current > 1` / `if current < 8` guards.
- Remaining newtypes (`NormGain`, `Seconds`, `LoopRegion`) deferred — out of
  scope for this task; would require changes to all signal read/write sites.

### Issue 5 — Clippy deny list
- Added `#![warn(clippy::unwrap_used)]` and `#![warn(clippy::todo)]` to
  `src/lib.rs`.
- All `#[allow(dead_code)]` and `#[allow(unused_imports)]` suppressions in
  state files annotated with explanatory comments.
